# Mini NPU Simulator

## 1. 프로젝트 소개
이 프로젝트는 입력된 패턴이 어떤 필터와 더 유사한지 MAC(Multiply-Accumulate) 연산으로 판별하는 Python 콘솔 애플리케이션이다.  
직접 3x3 필터와 패턴을 입력해 결과를 확인하는 모드 1과, `data.json`에 저장된 여러 크기의 필터/패턴을 일괄 분석하는 모드 2를 구현했다.  
또한 각 크기별 연산 시간을 측정하여 패턴 크기가 커질수록 연산량이 어떻게 증가하는지 확인할 수 있도록 구성했다.

### 3-2. MAC 연산 구현 개요

MAC 연산은 두 행렬의 같은 위치 원소끼리 곱한 뒤, 그 결과를 모두 더하는 방식으로 구현했다.

예를 들어 Matrix.mac(other)는 다음 순서로 동작한다.

1. 두 행렬의 크기가 같은지 확인한다.
2. 2중 반복문으로 각 위치의 값을 순회한다.
3. 같은 위치의 원소끼리 곱한다.
4. 곱한 값을 누적 합산한다.
5. 최종 합계를 점수로 반환한다.

즉, 패턴과 필터가 같은 위치에서 많이 겹칠수록 점수가 커지고, 더 유사한 패턴으로 판단할 수 있다.

### 3-3. 라벨 정규화 방식

문제 요구사항에 따라 프로그램 내부에서는 표준 라벨을 `Cross`, `X` 두 개만 사용한다.

JSON에는 여러 형태의 라벨이 들어올 수 있으므로, 로드 단계에서 라벨을 정규화했다.

정규화 규칙
- cross, Cross, + → Cross
- x, X → X

이 정규화는 다음 두 곳에 적용했다.

1) 필터 키 정규화

`filters`를 로드할 때 원본 키를 정규화해서 내부에서는 항상 `Cross`, `X`로 저장한다.

예:
```JSON
"size_5": {
  "cross": [...],
  "x": [...]
}
```

내부 변환 후:
```JSON
{
    5: {
        "Cross": Matrix(...),
        "X": Matrix(...)
    }
}
```

2) expected 값 정규화

`patterns`를 로드할 때 `expected`도 정규화한다.

예:

- "expected": "cross" → "Cross"
- "expected": "X" → "X"

이렇게 하면 이후 판정 결과와 비교할 때 문자열 표현 차이 때문에 FAIL이 나는 문제를 막을 수 있다.

### 3-4. 동점 처리 정책(epsilon)

점수 비교는 단순히 ==로 하지 않고 허용오차 epsilon을 사용하는 방식으로 처리했다.

```Python
abs(score_cross - score_x) < 1e-9
```

위 조건을 만족하면 두 점수를 사실상 같은 값으로 보고 UNDECIDED로 판정한다.

현재 입력은 주로 0, 1만 사용하므로 실제 오차가 크게 드러날 가능성은 낮다.
하지만 과제 요구사항에서 부동소수점 비교 정책과 허용오차 기반 비교를 명시하고 있으므로, 더 일반적인 수치 비교 상황까지 고려해 epsilon을 적용했다.

### 3-5. 예외 처리

JsonDataLoader에서는 다음과 같은 예외 상황을 검증하도록 구현했다.

- data.json 파일이 없는 경우
- JSON 형식이 잘못된 경우
- filters 또는 patterns 키가 없는 경우
- 필터 키 형식이 잘못된 경우 (size_5 형태 아님)
- 패턴 키 형식이 잘못된 경우 (size_5_1 형태 아님)
- cross/x 필터 또는 input/expected 값이 없는 경우
- 지원하지 않는 라벨이 들어온 경우

하위 계층인 로더에서는 예외를 발생시키고, 상위 계층인 MiniNpuSimulator에서 이를 받아 사용자에게 오류 메시지를 출력하도록 역할을 나누었다.

## 5. 결과 리포트
### 5-1. FAIL 케이스 원인 분석

모드 2에서 FAIL이 발생할 수 있는 대표적인 경우는 다음과 같다.

1) 라벨 표현 차이

정규화를 하지 않으면 cross, Cross, +가 서로 다른 문자열로 취급되어, 실제 의미는 같아도 FAIL이 발생할 수 있다.
이를 방지하기 위해 필터 키와 expected를 모두 Cross, X로 정규화했다.

2) 동점 처리

두 필터 점수가 epsilon 범위 내에서 같으면 UNDECIDED로 처리한다.
이 경우 expected는 Cross 또는 X 중 하나이므로, 자동 비교 시 FAIL로 집계된다.
즉, 동점은 계산 오류가 아니라 비교 정책에 따른 정상적인 FAIL 케이스다.

3) 크기 불일치

패턴의 크기와 필터의 크기가 다르면 MAC 연산 자체를 할 수 없다.
이 경우 프로그램이 비정상 종료되지 않도록 예외를 잡아서 해당 케이스를 FAIL로 처리한다.

4) JSON 스키마 문제

키가 누락되거나 형식이 잘못되면 정상적인 분석이 불가능하다.
이 경우 로더에서 예외를 발생시키고, 상위에서 오류 메시지를 출력하도록 했다.

### 5-2. FAIL이 0개라면

FAIL이 0개라면, 이는 다음 두 가지 정책이 올바르게 동작했기 때문이라고 설명할 수 있다.

- 라벨 정규화 정책
cross, Cross, +, x, X 같은 다양한 표현을 모두 내부 표준 라벨 Cross, X로 변환했기 때문에 문자열 표현 차이로 인한 오판정이 발생하지 않았다.
- 동점 비교 정책(epsilon)
두 점수가 사실상 같은 경우를 UNDECIDED로 정확히 분리해 처리했기 때문에, 부동소수점 비교 문제 없이 판정 기준이 일관되게 적용되었다.

즉, FAIL이 0개라는 것은 단순히 운이 좋았다는 뜻이 아니라, 정규화와 비교 정책이 일관되게 적용되어 데이터와 로직이 정확히 맞아떨어졌다는 의미로 해석할 수 있다.

## 6. 성능 분석
### 6-1. 성능 측정 방식

성능 측정은 입출력(I/O) 시간을 제외하고, 순수 연산 구간만 측정했다.

- time.perf_counter() 사용
- 각 크기별로 MAC 연산을 10회 반복
- 평균 시간을 ms 단위로 출력

성능 분석 표에는 다음 정보를 포함한다.

- 크기(NxN)
- 평균 시간(ms)
- 연산 횟수(N²)
  
### 6-2. 성능 표 해석

크기가 커질수록 평균 시간이 증가하는 이유는 MAC 연산이 행렬의 모든 위치를 한 번씩 확인해야 하기 때문이다.
예를 들어 5x5는 25칸, 13x13은 169칸, 25x25는 625칸을 순회해야 한다.
즉, 처리해야 할 데이터 수 자체가 크게 증가하고, 그에 따라 연산 시간도 증가한다.

3x3은 모드 2의 JSON 분석 대상은 아니지만, 성능 비교 기준을 보여주기 위해 코드 내부에서 샘플 필터/패턴을 생성해 함께 측정했다.

## 7. 시간 복잡도 분석

이 프로그램의 핵심 연산은 Matrix.mac()이다.
MAC 연산은 두 행렬의 같은 위치 원소끼리 곱하고, 그 결과를 모두 더하는 구조로 되어 있다.
행렬의 크기가 N x N이면 전체 원소 개수는 N²개이다.
따라서 MAC 연산은 모든 원소를 한 번씩 방문해야 하므로 기본적으로 O(N²)의 시간 복잡도를 가진다.
(시간복잡도란, 입력 크기가 커질 때 프로그램 실행에 필요한 연산량이 얼마나 늘어나는지를 나타내는 개념)
구체적으로는 바깥 반복문이 N번, 안쪽 반복문도 N번 실행되므로 총 반복 횟수는 N * N = N²가 된다.
각 반복에서 수행하는 연산은 배열 접근, 곱셈, 덧셈 정도이므로 상수 시간 작업으로 볼 수 있다.
즉, 전체 시간 복잡도는 O(N²)로 정리할 수 있다.
모드 2에서는 패턴 하나를 분석할 때 Cross 필터와 X 필터에 대해 각각 MAC 연산을 수행하므로, 엄밀히 보면 2 * O(N²)이지만 상수는 제거하므로 여전히 O(N²)이다.
성능 분석에서 3x3, 5x5, 13x13, 25x25 순으로 크기를 늘려가며 시간을 측정했을 때도, 처리 시간이 증가하는 경향을 확인할 수 있다.
이 증가 경향은 단순히 입력 파일이 커졌기 때문이 아니라, 실제로 순회해야 하는 셀의 개수가 N² 비율로 증가하기 때문이다.
예를 들어 5x5는 25칸이지만 25x25는 625칸이므로, 같은 방식의 MAC 연산이라도 훨씬 많은 위치를 검사해야 한다.
따라서 입력 크기가 커질수록 CPU가 수행해야 하는 반복 횟수가 커지고, 평균 실행 시간도 함께 증가한다.
이 점은 문제 설명에서 말하는 “대규모 MAC 연산이 CPU에서 병목이 되는 이유”와도 연결된다.
즉, 이 프로젝트는 작은 크기에서는 단순한 반복문으로도 충분하지만, 큰 크기로 확장될수록 O(N²) 구조 때문에 성능 부담이 빠르게 커진다는 점을 보여준다.
